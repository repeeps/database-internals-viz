수식·코드 없는 직관 입문 · 0번 + 16챕터

데이터베이스, 눈으로 보기

데이터가 쌓일수록 찾기가 느려지고, 여럿이 동시에 고치면 값이 깨져요. 데이터베이스 엔진은 이 둘을 어떻게 풀까요? 빨리(그리고 똑똑하게) 찾고(인덱스·실행계획), 안전하게 바꾸는(트랜잭션·MVCC) 보이지 않는 동작을 — SQL 문법이 아니라 엔진 속 그림으로 익히는 0번 입문 + 16챕터. (웹의 FastAPI 쿼리, 동시성 트랙의 잠금이 바로 이 아래에서 벌어지는 일이에요.)

👋 처음이세요?
0번 — 데이터베이스라는 문제 (찾기 느림 + 동시 수정 깨짐)
종이 장부가 두꺼워지면 왜 느려지고, 둘이 동시에 고치면 왜 깨질까. 엔진의 두 숙제 — 빨리 찾기 / 안전하게 바꾸기 — 를 한눈에. →

📚 처음이라면 0번부터 순서대로. 이건 SQL 강좌가 아니라 멘탈 모델이에요 — 디스크에서 페이지를 읽고, B+tree로 한 번에 찾아 들어가고, 플래너가 더 짧은 길을 고르고, 여러 트랜잭션이 같은 행을 동시에 건드려 값이 깨지고, 크래시가 나도 살아남는 걸 직접 보며 익혀요. 척추는 저장(A) → 인덱스(B) → 쿼리 실행(C) → 트랜잭션(D) → 내구성·확장(E). 인덱스(B)와 트랜잭션·동시성(D)이 두 기둥이에요.

PART A · 저장과 읽기
01

디스크는 느리다

메모리와 디스크의 속도 차, 페이지(블록) 단위 읽기, 순차 vs 랜덤, 버퍼풀.

pagebuffer pool
02

비용의 언어 · 풀 스캔

인덱스 없으면 처음부터 끝까지. 행 수에 비례하는 비용 — 모든 비용의 기준점.

full scancost
PART B · 인덱스
03

정렬의 힘

정렬돼 있으면 이진탐색(전화번호부), 아니면 못 한다. 인덱스 = 미리 정렬한 길잡이.

binary search
04

B+tree

왜 이진트리가 아니라 다진 트리인가 — 디스크 페이지 친화·얕은 깊이. 리프는 정렬 연결.

B+tree
05

범위 · 정렬에 쓰기

BETWEEN·ORDER BY·접두 매칭. 연결된 리프를 따라가며 인덱스가 정렬을 공짜로 준다.

rangeorder by
06

인덱스의 비용 · 모양

쓰기 느려짐·공간·카디널리티. + 인덱스가 행을 가리키나(secondary) 품나(clustered).

tradeoffclustered
PART C · 쿼리 실행
07

쿼리 플래너

같은 질문, 여러 실행 경로. 비용을 견줘 더 짧은 길을 고르는 옵티마이저 — 길 고르기 게임.

optimizercost
08

통계와 카디널리티

행 수 추정이 계획을 좌우한다. 추정이 어긋나면 인덱스를 두고도 풀스캔하는 참사.

statistics
09

조인

Nested Loop / Hash / Merge를 나란히 경주 — 입력 크기·정렬·인덱스로 승자가 바뀐다.

join
PART D · 트랜잭션과 동시성
10

트랜잭션 · ACID

전부 아니면 전무(원자성)·커밋/롤백. 한 덩어리로 취급한다는 약속.

ACIDatomic
11

동시 실행의 위험

dirty·non-repeatable·phantom read — 종이 장부로 직접 재현하는 세 가지 사고.

anomalies
12

락 · 격리수준

한 번에 하나만(락). 격리수준 4단계 × 이상현상 3종 전체 지도. 데드락도 여기.

lockisolation
13

MVCC

각자 다른 시간의 사진을 본다 — 읽기는 쓰기를 막지 않는다. 현대 DB의 심장.

MVCCsnapshot
PART E · 내구성과 확장
14

WAL · 내구성

로그 먼저, 데이터 나중. 크래시가 나도 로그로 되살린다(redo/undo).

WALrecovery
15

복제 · 샤딩

읽기를 나누고(복제) 데이터를 쪼갠다(샤딩). 일관성↔지연의 맞교환.

replicationsharding
16

너머 · 마무리

컬럼스토어·OLAP·NoSQL·캐시 한눈 지도. 그리고 다시 종이 장부로.
