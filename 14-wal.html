PART E · 내구성과 확장 — CH 14

WAL · 내구성

ACID의 마지막 글자 D(내구성): "커밋했으면 정전이 나도 사라지지 않는다". 그런데 변경된 데이터 페이지를 매번 디스크에 즉시 쓰는 건 너무 느려요(랜덤 쓰기). 그렇다고 메모리에만 두면 정전 때 날아가죠. 해법이 WAL(Write-Ahead Log, 미리 쓰는 로그)이에요. 핵심 규칙: 데이터를 바꾸기 전에, 무엇을 바꿀지 먼저 로그에 적는다. 로그는 순차로 줄줄 쓰는 작은 기록이라 빠르고, 커밋 = 로그를 디스크에 확실히 기록(flush)하는 순간이에요. 실제 데이터 페이지는 나중에 여유롭게 디스크로 내려가요. 그래서 정전이 나도 — 로그만 있으면 다시 켤 때 재생해서 복구해요: 커밋됐는데 데이터에 아직 안 반영된 건 다시 적용(redo), 커밋 안 했는데 데이터 페이지엔 흘러간 건 되돌리기(undo). "로그 먼저, 데이터 나중"이 내구성의 비밀이에요.

Step 00 — Crash After Commit

커밋했는데, 디스크엔 아직

변경은 일단 메모리(버퍼풀)에서 일어나요 — 빠르니까. 문제는 커밋 직후 데이터 페이지가 디스크로 내려가기 전에 정전이 나면? 버튼으로 비교하세요. 로그 없음: 메모리의 변경이 통째로 증발 → 커밋한 게 사라짐(내구성 깨짐). 로그 있음: 변경을 미리 로그에 적어 뒀으니, 다시 켤 때 로그로 되살려요. 데이터 페이지를 즉시 안 써도 안전한 거죠.

→ 데이터 페이지 디스크 쓰기는 느려서 미뤄요 — 대신 작은 로그를 먼저 안전하게.

Step 01 — Log First

작고 빠른 로그를 먼저 굳힌다

WAL의 순서를 보세요. ① 변경 내용을 로그에 순차로 적고(작고 빠름), ② 커밋 = 그 로그를 디스크에 flush(이 순간 내구성 확보), ③ 실제 데이터 페이지는 나중에 여유롭게 디스크로. 핵심은 "데이터보다 로그가 먼저 디스크에 닿는다" — 그래서 어느 시점에 죽어도 로그만 보면 뭘 해야 할지 알아요. 로그는 한 줄로 이어 쓰니(순차) 랜덤 데이터 쓰기보다 훨씬 싸요.

→ 커밋 = "로그가 디스크에 확실히 적힘". 데이터 반영은 그 뒤 천천히.

Step 02 — Recovery

다시 켤 때: redo와 undo

왜 두 방향의 복구가 필요할까요? 성능 때문에 엔진은 ① 커밋한 변경도 데이터 페이지엔 아직 안 내려갔을 수 있고(no-force), ② 커밋 안 한 변경이 데이터 페이지엔 먼저 흘러갔을 수도 있어요(steal). 그래서 복구 때 로그를 보고 두 가지를 해요. 버튼으로 보세요. redo: 로그엔 커밋됐는데 데이터엔 없는 변경 → 다시 적용. undo: 로그에 커밋 기록이 없는데 데이터엔 흘러간 변경 → 되돌리기. 이 둘이면 어떤 시점에 죽어도 깔끔히 복구돼요.

→ redo=커밋된 것 살리기, undo=미커밋 흘러간 것 지우기. 로그가 둘 다의 근거예요.

Step 03 — Takeaways

정리

WAL = "데이터보다 로그를 먼저 디스크에". 로그는 작고 순차라 빠르고, 커밋 = 로그 flush가 내구성을 보장해요. 실제 데이터 쓰기는 미뤄도 돼요 — 정전이 나면 로그를 재생해 redo(커밋된 것 다시 적용)·undo(미커밋 흘러간 것 되돌림)로 복구하니까요. (로그가 무한히 길어지지 않게 주기적으로 체크포인트로 "여기까진 데이터에 반영됨"을 찍어요.) 이로써 한 대의 데이터베이스 이야기를 마쳐요. 다음엔 한 대를 넘어 — 복제와 샤딩으로 규모를 키워요.
