# 데이터베이스, 눈으로 보기 — 목차 초안 (리뷰용)

## 도메인·결
- **무엇**: 데이터베이스 **엔진 내부**의 직관. "디스크 위 거대한 데이터를 ① 어떻게 빨리 찾고(인덱스·쿼리실행) ② 어떻게 여럿이 동시에 안전하게 바꾸나(트랜잭션·MVCC)". SQL 문법 강좌 아님 — **보이지 않는 엔진 동작**을 멘탈 모델로.
- **결**: 다른 트랙처럼 "수식·코드 없는 직관". SQL은 illustration으로만(개념 보조). 모든 동작(B-tree 탐색·스캔·락 타임라인)은 브라우저 실시간.
- **자매편**: **FastAPI**(쿼리·N+1·커넥션풀의 그 아래 계층) · **동시성**(잠금·경쟁·데드락 → 트랜잭션 격리수준으로 확장) · **알고리즘**(이진탐색·B-tree·해시·정렬·조인) · **분산**(복제·샤딩 예고).
- **두 하이라이트**: **B 인덱스**(왜 빠른가, B-tree) · **D 트랜잭션·동시성**(격리수준·MVCC — 동시성 트랙의 DB 버전).

## 엔진
- 베이스 lib.js = **VZ.AL**(스텝 플레이어) — DB 동작은 이산 이벤트(B-tree 한 노드씩 탐색, 트랜잭션 한 연산씩)라 슬라이더 아닌 스텝 플레이어가 결. 알고리즘/ROS2/FastAPI와 동일 베이스 + VZ.LA.tween.
- 신규 **VZ.DB**: page/buffer(페이지·버퍼풀)·table+rows(스캔 애니)·**btree**(노드·키, 탐색경로 하이라이트, 분기)·**planTree**(쿼리 실행계획 트리)·**txTimeline**(트랜잭션×시간, 락 보유/대기 — 동시성 timeline 재활용)·lockMatrix(공유/배타 호환)·**versionChain**(MVCC 버전 사슬·스냅샷)·walStrip(로그 선기록)·costBar.

## 구성 — 0번 입문 + 16챕터 (5 PART)

**0번 · 데이터베이스라는 문제**
종이 장부 비유 — 행이 쌓이면 찾기 느림, 둘이 동시에 고치면 깨짐. 두 핵심: **빨리(그리고 똑똑하게) 찾기**(인덱스·실행계획) + **안전하게 바꾸기**(트랜잭션). "찾는 길이 여럿일 때 누가 고르나?"를 던져 PART C도 훅 안으로. 전체 지도 + 용어표(테이블·행·페이지·인덱스·트랜잭션). *(리뷰 M1: 훅에 '똑똑하게 찾기' 추가해 C파트 내러티브 누수 차단)*

**PART A · 저장과 읽기 (기초)**
- 01 디스크는 느리다 — 메모리 vs 디스크 속도차, **페이지(블록) 단위**로 읽음, 순차 vs 랜덤(랜덤이 비쌈 — HDD는 탐색, SSD는 페이지당 오버헤드), 버퍼풀(캐시) *(리뷰 M3: SSD 시대에도 evergreen하게)*
- 02 비용의 언어 · 풀 테이블 스캔 — 인덱스 없으면 처음부터 끝까지, **행 수에 비례**. 여기서 **costBar(비용 막대)와 "모든 비용의 기준점(baseline)" 직관**을 도입(C파트 복선) *(리뷰 M4: 02를 '비용의 언어'로 격상해 단독 챕터 값 확보)*

**PART B · 인덱스 [하이라이트]**
- 03 정렬의 힘 — 정렬돼 있으면 이진탐색(전화번호부), 안 됐으면 못 함. 인덱스 = "미리 정렬해 둔 길잡이"
- 04 B-tree — 왜 이진트리가 아니라 다진(多枝) 트리인가(디스크 페이지 친화·얕은 깊이). **B+tree를 실제 구조로 확정**: 데이터는 리프에, 리프끼리 정렬 연결(05 범위검색의 전제). 루트→리프 탐색 애니 *(리뷰 H1: '여부 명시' 애매함 제거, B+tree 확정)*
- 05 범위·정렬에 쓰기 — BETWEEN·ORDER BY·접두 매칭, 연결된 리프를 따라가며 인덱스가 정렬을 공짜로 제공
- 06 인덱스의 비용 + 인덱스의 모양 — 쓰기 느려짐·공간·카디널리티 낮으면 무용·과잉 인덱스("공짜 아님"). + **인덱스가 행을 *가리키나*(secondary) 행을 *품나*(clustered), 커버링 인덱스가 추가 조회를 없앤다** *(리뷰 H4: clustered vs secondary 흡수 — 인덱스 파트 최대 실무 갭)*

**PART C · 쿼리 실행**
- 07 쿼리 플래너 — **"넌 이미 두 경로를 봤다"**(풀스캔 vs 인덱스 스캔)로 다리 놓기. 같은 질문, 여러 실행 경로 → costBar로 더 짧은 길 고르는 **"길 고르기" 게임**(직접 골라보고 엔진 선택과 비교) *(리뷰 H1: C 진입 절벽 완화)*
- 08 통계와 카디널리티 — 메커니즘보다 **"추정 오류의 재앙"**: 분포 편향 슬라이더 → 추정 막대 vs 실제 막대 어긋남 → 07 costBar 오판 → **인덱스 두고 풀스캔 고르는 참사**를 눈으로 *(리뷰 H2: 순수 비주얼로 구출)*
- 09 조인 — **"조인 레이스"**: Nested Loop / Hash / Merge 셋을 나란히 돌리고 입력 크기 슬라이더로 **결승 순서가 바뀜**(Hash=대용량 비정렬·equi-join 한정, Merge=정렬됨, NL=작은쪽+인덱스) *(리뷰 M3·L2: 과밀을 경주로, equi-join 한정 명시)*

**PART D · 트랜잭션·동시성 [하이라이트]**  *(리뷰 M2: 11~13 각 첫 스텝은 종이 장부로 자력 재현 — 동시성 트랙 없이 이해, 자매 연결은 '더 깊이' 배지로만)*
- 10 트랜잭션·ACID — **A·I 중심**(원자성=전부 아니면 전무·커밋/롤백, 격리는 11~13 예고). D(내구성)는 "14에서", C는 가볍게 *(리뷰 L4)*
- 11 동시 실행의 위험 — dirty·non-repeatable·phantom를 **종이 장부로 자력 재현**(A가 확정 안 한 숫자를 B가 읽고 행동→A 취소). 끝에 "고치는 법=격리수준(다음 장)" 예고. 동시성 *경쟁 상태*의 **트랜잭션판 사촌**(동일시 아님) *(리뷰 C2·M4)*
- 12 락·격리수준 — 공유/배타 락·2PL·데드락(동시성 *데드락* 회수). **핵심 인터랙션=격리수준 4×3 그리드**(Read Uncommitted→Committed→Repeatable Read→Serializable × dirty/non-repeatable/phantom). **Serializable은 셋 다 차단**(phantom만 아님). 락은 격리의 *한 구현* *(리뷰 C1)*
- 13 MVCC — **"각자 다른 시간의 사진"**: 두 트랜잭션 말풍선이 서로 다른 버전을 가리킴. "읽기는 쓰기를 안 막는다"(읽기-쓰기 통과/쓰기-쓰기 충돌). + **버전이 쌓이면 청소(GC) 필요**(공짜 아님). 현대 DB의 심장 *(리뷰 H3)*

**PART E · 내구성·복구·확장**
- 14 WAL·내구성 — "로그 먼저, 데이터 나중". **steal/no-force**(커밋 전 더티페이지가 디스크로 감)를 먼저 깔아 redo(커밋했지만 미반영)·undo(미커밋인데 디스크 감)가 *왜 필요한지* 성립. 체크포인트("D" in ACID) *(리뷰 H2)*
- 15 복제·샤딩 — 읽기 분산(복제, 동기/비동기=일관성↔지연·읽은-내-쓰기 staleness)·쪼개기(샤딩=조인/트랜잭션 어려워짐). 분산 일관성은 분산 트랙 예고 한 줄 *(리뷰 L2·L3: 두 장면으로 압축)*
- 16 너머·마무리 — **수미상관**: 0번 종이 장부로 돌아가 "이제 이 안에서 무슨 일이 벌어지는지 다 보이죠". 풍경 지도(컬럼스토어/OLAP·NoSQL·캐시는 *나열* 아닌 한눈 지도) + FastAPI/동시성/알고리즘 연결 착지 *(리뷰 M2·L3: 백화점식 지양, 정서적 payoff)*

## 챕터 구조 (전 트랙 공통)
4스텝: 직관 → 인터랙션 → 심화/규칙 → 정리(+다음 예고). 다크 neural-lab 테마, 허브 "처음이세요?" 배너.

## 경계 (다른 트랙·도메인에 위임)
- 분산 합의(Raft)·CAP 깊이 → 분산 시스템 트랙(예정)에 위임, 여기선 복제/샤딩 *맛보기*만
- SQL 문법·쿼리 작성법은 강좌 안 함(개념 보조로만 SQL 노출)
- 특정 DBMS 구현 세부(PostgreSQL vs MySQL 차이)는 안 감 — 일반 원리 수준
- 동시성 *원리*(경쟁/잠금/데드락 자체)는 동시성 트랙, 여기선 그 *DB 적용*(격리수준·MVCC)

## 빌드 시 정확성 가드레일 (리뷰 반영)
- 01: 랜덤 접근이 순차보다 비쌈 — HDD는 **탐색(seek) 지연**, SSD는 **페이지당 오버헤드·읽기 증폭**(대역폭만의 문제 아님). 페이지 단위 I/O가 durable core
- 04: **B+tree로 확정** — 균형·다진·정렬, 데이터/포인터는 리프에, **리프끼리 정렬 연결**(이게 05 범위검색을 가능케 함). 깊이 O(log_f n), 디스크 읽기 횟수 = 깊이
- 06: 인덱스는 **읽기 빠름 ↔ 쓰기/공간 비용** 트레이드오프, 낮은 카디널리티 무용. + **clustered(행을 품음) vs secondary(행을 가리킴)**, 커버링 인덱스가 추가 조회 제거
- 09: 조인 승자는 입력 크기·정렬·인덱스에 의존(Hash=대용량 비정렬·**equi-join 한정**, Merge=정렬됨, NL=작은쪽+인덱스)
- 11: dirty·non-repeatable·phantom는 *논리적 읽기 이상*(바이트 단위 data race 아님) — 경쟁상태의 **사촌**이지 동일물 아님
- 12: **격리수준 4×3 전체 매핑**(RU=dirty허용 / RC=dirty차단·non-repeatable허용 / RR=non-repeatable차단·phantom허용 / Serializable=**셋 다 차단**). "Serializable=phantom만 차단"은 오류. 락은 격리의 *한 구현*
- 13: MVCC=읽기 안 막힘(스냅샷)이지 잠금 전무 아님(**쓰기-쓰기는 여전히 충돌**). 버전 누적→**GC(vacuum) 필요**(공짜 아님). 동시성 "가시성"과 연결
- 14: WAL="로그 먼저, 데이터 나중" + **steal/no-force** 전제 — 더티페이지가 커밋 전 디스크로 갈 수 있어 **undo**, 커밋됐지만 미반영이라 **redo**. 커밋=로그 flush
- 15: 복제는 일관성↔지연(동기/비동기, read-your-writes staleness), 샤딩은 조인·트랜잭션이 어려워짐 — 만능 아님

## 리뷰 통과 기록
- **목차 리뷰 2종(도메인 전문가 + 학습설계/초심자) 반영 완료**. 평결: 골격·순서·두 하이라이트·범위 경계 견고 → 빌드 가능. 정확성 가드레일 3건(격리 4×3·B+tree 확정·WAL steal/no-force)과 학습 길목 2곳(C 진입 다리·MVCC/카디널리티 비주얼) 보강. 구조는 0+16·5PART 유지(슬롯 추가 없이 CH02 격상·CH06 흡수로 갭 메움).
