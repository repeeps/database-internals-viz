PART B · 인덱스 — CH 05

범위 · 정렬에 쓰기

인덱스가 단순히 "한 값 콕 찾기"에만 좋은 건 아니에요. B+tree의 정렬 연결된 리프 덕분에 훨씬 더 많은 걸 공짜로 얻어요. ① 범위 검색("30 이상 60 이하"): 시작 값이 든 리프를 한 번 찾은 뒤, 옆으로 연결된 리프를 죽 따라가면 끝값까지 정렬된 채로 줄줄 나와요. ② 정렬(ORDER BY): 인덱스는 이미 정렬돼 있으니, 정렬 단계를 통째로 건너뛰고 인덱스 순서대로 읽기만 하면 돼요. ③ 복합 인덱스(여러 열을 묶은 인덱스, 예: 성+이름): 사전이 성으로 먼저 정렬되듯, 왼쪽 열부터 써야 효력이 있어요 — 성만으로도, 성+이름으로도 빠르지만 이름만으로는 못 써요(좌측 접두 규칙). 같은 인덱스라도 질문의 모양에 따라 쓸 수 있고 없고가 갈려요.

Step 00 — Range Scan

시작점 하나 찾고, 옆으로 죽

"키가 30 이상 60 이하인 행"을 찾아요. ▶로 보세요 — 먼저 트리를 타고 내려가 시작값(30)이 든 리프를 찾고(딱 한 번의 트리 탐색), 그다음은 연결된 리프를 옆으로 따라가며 60을 넘을 때까지 줄줄 읽어요. 풀스캔처럼 전부 보지 않고, 필요한 구간만 정확히 훑죠. 굵게 표시된 리프가 읽은 범위예요.

→ 범위 검색 = 트리 탐색 1번 + 리프 연결 따라가기. B+tree의 "+"(리프 연결)가 빛나는 순간.

Step 01 — Sort for Free

정렬을 건너뛴다

"이름순으로 정렬해서 보여줘"(ORDER BY). 버튼으로 비교하세요. 인덱스 없음: 데이터를 다 모은 뒤 따로 정렬하는 단계가 필요해요(시간·메모리 듦). 인덱스 있음: 인덱스가 이미 그 순서로 정렬돼 있으니, 정렬 단계를 통째로 건너뛰고 인덱스를 순서대로 읽기만 하면 끝. 정렬이 공짜가 돼요.

→ 인덱스 순서 = 원하는 정렬 순서이면, "정렬" 연산 자체가 사라져요.

Step 02 — Composite · Left-Prefix

복합 인덱스는 왼쪽부터

(성, 이름)으로 묶은 복합 인덱스가 있어요 — 전화번호부처럼 성으로 먼저 정렬되고, 같은 성 안에서 이름으로 정렬돼요. 각 질문을 눌러 쓸 수 있는지 보세요. 성='김': ✓(맨 앞에서 모임). 성='김' 그리고 이름='철수': ✓(접두 그대로). 이름='철수'만: ✗ — 이름만으론 인덱스 전체에 흩어져 있어 못 써요(전화번호부에서 이름만으론 못 찾듯). 왼쪽 열부터 이어 써야 효력이 있어요.

→ 좌측 접두 규칙: (A,B,C) 인덱스는 A / A,B / A,B,C엔 쓰지만 B나 C 단독엔 못 써요.

Step 03 — Takeaways

정리

정렬된 연결 리프 덕에 인덱스는 한 값 찾기를 넘어 범위 검색(시작점 + 옆으로 따라가기)과 정렬 건너뛰기(이미 정렬됨)까지 줘요. 복합 인덱스는 왼쪽 열부터 써야 효력이 있고요(좌측 접두). 같은 인덱스도 질문의 모양에 따라 쓰이고 안 쓰여요 — 그래서 "어떤 인덱스를, 어떤 순서로" 만들지가 설계의 핵심이에요. 그런데 인덱스는 공짜가 아니에요. 다음엔 그 대가를 봐요.
