PART B · 인덱스 — CH 03

정렬의 힘

풀 스캔이 느린 건 데이터가 아무 순서 없이 쌓여 있어서예요 — 어디 있는지 모르니 다 봐야 하죠. 그런데 데이터가 정렬돼 있다면 완전히 다른 게임이 돼요. 전화번호부에서 '김'씨를 찾을 때 1쪽부터 넘기지 않잖아요 — 중간을 펴 보고, 앞쪽이면 앞 절반만, 뒤쪽이면 뒤 절반만 다시 봐요. 이게 이진 탐색이고, 볼 때마다 후보가 절반으로 줄어 100만 개도 단 스무 번이면 찾아요(로그 시간). 문제는 실제 테이블은 정렬돼 있지 않다는 거예요(들어온 순서대로 쌓이니까). 그래서 엔진은 데이터를 직접 정렬하는 대신, "정렬된 길잡이"를 따로 만들어 둬요 — 그게 바로 인덱스예요. 책 뒤의 찾아보기(색인)처럼, 정렬된 키와 "그 행이 어디 있는지"만 담은 별도 목록이죠.

Step 00 — Sorted vs Unsorted

정렬돼 있으면 반씩 버릴 수 있다

버튼으로 비교하세요. 무정렬: 값이 뒤섞여 있어 중간을 봐도 어느 쪽에 있는지 알 수 없어요 — 결국 다 봐야 해요(풀 스캔). 정렬: 중간 값만 보면 찾는 값이 그보다 큰지 작은지 알 수 있어, 한쪽 절반을 통째로 버려요. 정렬 하나로 "다 보기"가 "반씩 줄이기"로 바뀌어요.

→ 정렬은 "중간만 봐도 방향을 안다"를 가능케 해요 — 이게 이진 탐색의 전제예요.

Step 01 — Binary Search

볼 때마다 후보가 절반

정렬된 값에서 37을 찾아요. ▶로 한 단계씩 보세요 — 가운데를 보고(중간값), 찾는 값과 견줘 왼쪽 절반이나 오른쪽 절반을 버려요. 회색이 버려진 범위예요. 16개를 단 4~5번 만에 좁히죠. 값이 1024개여도 10번, 100만 개여도 20번이면 끝 — 매번 반으로 줄기 때문이에요(로그 시간).

→ 풀스캔 O(n)이 이진탐색 O(log n)으로 — 100만→20. 이게 인덱스가 주는 마법이에요.

Step 02 — Index = Sorted Guide

원본은 그대로, 길잡이만 정렬

실제 테이블은 들어온 순서로 쌓여요(정렬 안 됨). 매번 통째로 정렬할 순 없죠. 그래서 엔진은 인덱스를 따로 둬요 — 정렬된 키와 "그 행이 원본 어디에 있는지"(포인터)만 담은 별도 목록이에요. 책 뒤 찾아보기(색인)와 똑같아요: 단어는 가나다순, 옆에 쪽 번호. 찾을 땐 정렬된 인덱스에서 이진 탐색 → 포인터로 원본 행으로 점프.

→ 인덱스는 원본의 정렬된 사본(키+포인터) — 원본은 안 건드리고 빠른 길만 추가해요.

Step 03 — Takeaways

정리

정렬은 "중간만 봐도 방향을 안다"를 가능케 해, 풀스캔(다 보기)을 이진 탐색(반씩 줄이기)으로 바꿔요 — O(n)에서 O(log n)으로. 하지만 원본 테이블은 정렬돼 있지 않으니, 엔진은 정렬된 키 + 포인터만 담은 별도 길잡이 = 인덱스를 둬요. 다음엔 이 정렬된 길잡이를 디스크에 맞게 만든 실제 자료구조 — B+tree — 를 봐요.
